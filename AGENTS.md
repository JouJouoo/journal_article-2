# AGENTS.md

本文件是本仓库的执行手册。默认用中文与用户沟通；代码、命令、文件名、API 名称保留原文。目标是把任务可靠完成：先读上下文，再做最小必要修改，最后用合适的方式验证。

## 1. 先判定任务类型

开始前先说明本次任务属于哪一类，并只执行对应检查项：

- `代码/脚本修改`: 读相关模块和测试，实施最小改动，运行相关测试。
- `实验运行`: 先确认实验协议、数据来源、训练规模和输出目录，再按阶梯执行。
- `标准算例/论文证据`: 先验证 IEEE 33/69 来源、尺度、约束和指标口径，再生成结论。
- `论文/图表/文档`: 先确认源文件和目标用途，独立检查图表或文档质量。
- `工作区清理`: 先列出保留项和删除/移动项，不把正式论文证据当作临时输出处理。
- `Git 操作`: 先看工作区状态和提交范围，不默认提交全部文件。

如果只是解释、审阅、查状态或改一个小文档，不要套用正式实验流程。

## 2. 工作区安全

- 修改前查看相关文件和现有模式；不要凭文件名猜测实现。
- 工作区可能已有用户改动。不要回滚、覆盖或格式化无关文件。
- 删除或移动文件前，确认目标都在当前仓库内；涉及 `outputs/`、论文文档或大目录时，先说明保留/清理边界。
- 不执行 `git reset --hard`、`git checkout --`、大范围删除或跨目录写入，除非用户明确要求。
- PowerShell 出现 `profile.ps1 cannot be loaded` 时，先看实际命令是否成功，不要仅凭该提示判失败。

## 3. 目录边界

- `src/tecsf/`: 可复用模型、环境、市场、链、指标和 benchmark 逻辑。
- `scripts/`: 命令行入口和实验编排。脚本应尽量薄，核心逻辑放回 `src/tecsf/`。
- `configs/`: 默认配置和实验协议配置。
- `data/`: 小型、必要、可追溯的数据源，例如 `data/case69.m`。
- `docs/`: 论文、实验协议、阶段报告和文献材料。
- `outputs/`: 生成的实验产物，默认被 Git 忽略。只保留仍需本地核查的论文级结果集。
- `tests/`: 单元测试、回归测试和 smoke 测试。

不要把一次性分析代码混进核心模块；不要把大型实验输出提交为源码。

## 4. 实验规则

只有“新的正式实验”或“用于论文结论的实验”必须先写 1 页实验协议。协议至少包含：

- 研究问题
- 数据来源，是否为官方标准算例
- 对比方法
- 随机种子
- 训练轮数和评价轮数
- 核心指标和验收阈值
- 该实验允许支持的论文结论

执行顺序默认是：

1. `smoke test`
2. `100/200 episode` 短训
3. `1000 episode` 正式实验

执行训练任务时，先根据 `episodes`、`horizon`、seed 数、variant 数、设备和历史耗时粗略估算运行时间；长时间任务只按能及时发现报错的最小必要频率轮询，进程正常运行且无报错时减少状态汇报，不启动无关实验或检查。

除非用户明确要求或协议已经确认，不要直接运行全量实验。普通单元测试、`--help`、导入检查、脚本 `--validate-only` 和临时 debug 不需要实验协议。

## 5. 标准算例和论文结论

- 使用 IEEE 33/69 做论文级实验前，先检查动作尺度、负荷尺度、线路容量、储能边界、可行性投影和指标口径。
- 论文级 IEEE 33/69 来源检查优先运行：

```powershell
& "C:\Users\zrway\.conda\envs\DP-LCRL\python.exe" scripts\validate_benchmark_sources.py --case69-m data\case69.m --strict
```

- `synthetic33` 只能表述为派生/合成 profile，不能写成官方 IEEE 33 标准算例。
- 论文或摘要里的性能结论必须能追溯到代码、测试、验证脚本或 `outputs/` 中的正式结果。不要把安全/结算机制证据扩写成全指标 SOTA 或普遍最优。
- 如果当前输出和严格来源验证不一致，先说明版本差异，再决定是否重跑。

## 6. 验证要求

- 代码或脚本改动后，运行最相关测试；触及共享逻辑、训练流程或论文指标时，优先运行完整测试：

```powershell
& "C:\Users\zrway\.conda\envs\DP-LCRL\python.exe" -m pytest
```

- 提交前或较大改动后运行 `git diff --check`。
- 文档-only 改动不强制跑测试，但要检查文件内容、路径和引用是否正确。
- 图表先导出独立 `png`/`pdf`，检查图例、坐标轴、字体、线条、颜色和标注，再插入 Word/docx。
- 如果无法验证，明确说明原因、影响范围和剩余风险。

## 7. Git 规则

提交或暂存前必须：

- 查看 `git status`
- 检查 `.gitignore`
- 区分源码、论文文件、配置、脚本和实验输出
- 只处理用户要求的文件范围
- 不使用 `git add .` 打包大型生成物或无关改动

实验输出默认不提交；如必须提交，只提交经过确认的报告、摘要或小型可追溯数据。

## 8. 沟通方式

- 中间更新简短说明正在做什么、发现了什么、下一步是什么。
- 遇到需求、代码状态和验证结果冲突时，先说明冲突点。
- 最终回复说明改了哪里、验证了什么、还剩什么风险。
- 不用冗长方案替代执行；除非用户要求只讨论，否则应推进到可验证结果。
